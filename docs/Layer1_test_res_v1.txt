## 失败模式分析

### 1. 路由器单路由 vs. 多 Agent 预期（主因，贡献约 35 个失败）

几乎所有 cross_domain、long_multipart、ambiguous 类别的查询都预期同时触发
2~3 个 Agent（如 `['training', 'rehab', 'nutrition']`），但实测 `agent_used`
字段始终只返回一个 Agent。例：

  adv-001: Expected ['training', 'rehab', 'nutrition'], Actual ['rehab']
  adv-004: Expected ['training', 'rehab', 'nutrition'], Actual ['rehab']
  adv-041: Expected ['training', 'rehab', 'nutrition'], Actual ['training']

根本原因：后端 Supervisor 对每次查询只派发一个 Agent，不支持多 Agent 并发
分发。此外，layer1_runner.py 的 query() 方法也只从 done 事件中读取单个
agent_used 字段，即使后端有多 Agent 支持，测试框架本身也无法捕获。

### 2. Rehab Agent 分配偏重（约 28 个失败显示 Actual: ['rehab']）

在所有单 Agent 路由结果中，rehab Agent 占据绝大多数。即便查询的核心是
training 或 nutrition（如 adv-006、adv-021、adv-041），路由器仍倾向于选择
rehab。可能原因：
  - Rebuilding Milo 被分块最细、chunks 数量最多，RAG 召回率更高
  - 查询中的损伤/疼痛关键词权重高，触发 rehab 优先

### 3. 语料库外查询未正确拒绝（4 个失败）

adv-023、adv-026、adv-030、adv-035 均属于 out_of_corpus 类别，预期
`expected_agents: []`（即系统应回答"语料库中无此内容"而不路由到任何 Agent），
但实测全部被路由到某个 Agent 并给出了回答。路由器缺乏"无法回答"的兜底路径。

### 4. 部分双 Agent 预期查询仅差一个 Agent（细节失败）

adv-020: Expected ['nutrition', 'training'], Actual ['nutrition']
adv-021: Expected ['training', 'nutrition'],  Actual ['training']
adv-040: Expected ['nutrition', 'training'],  Actual ['nutrition']

路由方向正确，只是未触发第二个 Agent，属于多路由支持缺失的子集。

## 回答质量观察

尽管路由失败率高，RAG 检索与回答生成质量整体尚可：
- 系统能正确识别语料库中没有相关内容，并如实告知（未编造答案）
- 多数回答会引用具体书名和章节，来源透明
- 中英混合查询（mixed_language 类别）均能以中文作答，语言对齐正常

## 主要问题优先级

P0 — 路由器不支持多 Agent 并发派发：若要使 cross_domain 类查询通过，需后端
     Supervisor 支持多路由，并更新 layer1_runner.py 的 query() 方法以收集
     多个 agent_used 值。

P1 — Rehab Agent 路由偏重：需审查 embedding 召回和路由决策逻辑，可能需对
     training/nutrition 相关关键词提高权重或对语料库 chunk 分布进行均衡。

P2 — Out-of-corpus 查询无兜底：路由器需增加"语料库无相关内容"分支，而非
     强制派发至某个 Agent。

P3 — layer1_runner.py 仅捕获单个 agent_used：即使后端修复多路由，测试框架
     也需同步更新以支持收集和比较多 Agent 结果。
